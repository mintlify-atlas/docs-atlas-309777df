## Introduction

The Gestión de Gastos Mibanco application follows a **layered architecture** (Clean Code principles) that ensures clear separation of concerns, maintainability, and testability. The system is designed around the **Single Responsibility Principle (SRP)** with dependency injection to facilitate unit testing.

<Note>
This architecture was designed for a technical challenge for the Software Engineering Internship position at Mibanco, demonstrating professional-grade Java development practices.
</Note>

## Core Design Principles

### Dependency Injection

The application uses **constructor-based dependency injection** to enable testability and loose coupling:

```java
public class GastoService {
    private final GastoRepository repository;

    // Dependency injection through constructor
    public GastoService(GastoRepository repository) {
        this.repository = repository;
    }
}
```

This design allows the service layer to work with any repository implementation, making it easy to:
- Swap implementations (e.g., file storage to database)
- Create fake repositories for testing
- Mock dependencies in unit tests

### Interface-Based Contracts

The repository layer uses interfaces to define contracts:

```java
public interface GastoRepository {
    List<Gasto> findAll();
    Optional<Gasto> findById(UUID id);
    void save(Gasto gasto);
    void update(Gasto gasto);
    void deleteById(UUID id);
}
```

This abstraction enables:
- Multiple implementations (currently `FileGastoRepository`)
- Easy testing with fake implementations
- Future extensibility (database, cloud storage, etc.)

### Single Responsibility Principle

Each layer has a clearly defined responsibility:

- **Model**: Represents domain entities
- **Repository**: Handles data persistence
- **Service**: Implements business logic and validation
- **UI**: Manages user interaction

<Note>
The separation of concerns ensures that changes in one layer don't ripple through the entire application, making the codebase more maintainable and easier to test.
</Note>

## Data Flow

The typical data flow for a CRUD operation follows this pattern:

1. **User Input** → UI Layer captures and validates format
2. **Service Invocation** → UI calls appropriate service method
3. **Business Validation** → Service layer validates business rules
4. **Repository Operation** → Service delegates persistence to repository
5. **File Persistence** → Repository serializes data to JSON using Jackson
6. **Response** → Success/error flows back through layers to user

Example flow for adding a new expense:

```
User enters expense data
       ↓
Consola.agregarGasto() validates format
       ↓
GastoService.agregarGasto() validates business rules
       ↓
Creates new Gasto with UUID
       ↓
FileGastoRepository.save() persists to JSON
       ↓
Success message displayed to user
```

## Error Handling Strategy

The application uses a custom exception hierarchy to handle errors gracefully:

```java
public class GastoException extends RuntimeException {
    public GastoException(String mensaje) {
        super(mensaje);
    }
}
```

Exceptions are caught at the UI layer to prevent application crashes and provide user-friendly error messages. See [Exception Handling](/architecture/technical-decisions) for details.

## Key Architectural Benefits

<Accordion title="Testability">
  Dependency injection and interface-based design enable comprehensive unit testing. The test suite uses fake repository implementations to isolate business logic from file I/O operations.
</Accordion>

<Accordion title="Maintainability">
  Clear separation of concerns means changes in one layer don't affect others. For example, switching from file storage to a database only requires implementing a new repository class.
</Accordion>

<Accordion title="Extensibility">
  The modular design makes it easy to add new features. New repository implementations, additional validation rules, or alternative UI interfaces can be added without modifying existing code.
</Accordion>

<Accordion title="Reliability">
  Custom exception handling and financial-grade precision (BigDecimal) ensure the application behaves predictably and handles edge cases gracefully.
</Accordion>
